###### CI e CD
**CI = Continuous Integration**
È la fase dedicata a:
* compilare il codice
* eseguire test automatici
* analizzare la qualità
* creare artefatti (immagini Docker)
CI risponde alla domanda: “Il codice si integra e funziona?”

Esempi di step CI (come i comandi) : 
```
build
npm install //(visto per i progetti vite)
dotnet build
```

**CD = Continuous Deployment**
È la fase dedicata a: 
* rilasciare automaticamente le versioni 
* distribuire in ambienti (dev, test)
* applicare configurazioni
* aggiornare container/pod
CD risponde alla domanda: “Posso distribuire questa versione in sicurezza?”

**Come si legano alle pipeline**
Le pipeline CI/CD sono *pipeline di automazione*, tipicamente descritte in YAML.
Quindi: 
* CI builda e testa.
* CD distribuisce.
* Le pipelines sono lo strumento che esegue CI/CD.

ci siamo focalizzati sulle pipelines, che in particolare possono essere gestite scritte sottoforma di file YAML.

#YAML = YAML Ain’t Markup Language
È un linguaggio leggibile, usato per configurazioni (come #Kubernetes e #AzurePipelines). Funziona con indentazione, senza parentesi.

anche i manifest Kubernetes (container da distribuire) sono scritti in yaml.

###### kubernetes
orchestratore di container.
abbiamo visto poi che vengono creati dei cotentitori (come delle scatole).
solo nel progetto di teseo si utilizza un Okd... perche è puu grande??
**Perché si arriva ad avere bisogno di OKD (o OpenShift)?** (come nel caso del progetto)
OKD viene introdotto quando la complessità di deploy, gestione dei container e orchestrazione supera quello che un semplice Docker o un cluster Kubernetes “puro” può offrire facilmente.
**L’app cresce e serve containerizzare**
All'inizio l’app gira su VM (virtual machine) o server fisici.
Si passa ai container (Docker) per:
* isolare l’app
* evitare problemi di funzionamento solo su alcuni pc
* gestire dipendenze 
* automatizzare build e deploy
Talvolta Docker da solo non basta.
**Ci sono troppi container da gestire**
Quando inizi ad avere:
* più microservizi
* più container
* più ambienti
* più istanze
…serve uno strumento che gestisca:
* scaling automatico
* distribuzione delle app
* restart automatici
Questo strumento è #Kubernetes.
Kubernetes risolve molto, ma è difficile da amministrare. al suo interno ci sono dei file (yaml), tra cui #configmap, deployment, ingress, pv e pvc (persistent volumes), e svc (servizi).
nel caso del progetto VVF (vigili del fuoco) si usa Kubectl.

###### OKD
#OKD (e la versione enterprise OpenShift) risolvono proprio questi problemi.
OKD fornisce: Kubernetes già configurato:
* rete
* registry interno
* monitoring
* security
CI/CD integrato
Non devi integrare strumenti esterni obbligatoriamente.
 **Un unico ambiente pronto per tutto il ciclo di vita**
* build
* deploy
* routing
* scaling
* monitoring
In pratica:
OKD è Kubernetes + tutto ciò che serve per usarlo in produzione stabilmente e in modo sicuro.

**Perché  OKD?**
perche per Kubernetes 
-  serve troppo tempo per gestirlo
-  la sicurezza è complicata
- mancano strumenti nativi per build e deploy
- aggiornamenti e manutenzione sono complessi

OKD risolve tutti questi problemi in un’unica piattaforma.
